# Escalation Playbook — 客户升级应对

客户升级不是你一个人的危机——是整个公司的危机处理能力测试。
好的升级处理能让客户更信任你（"他们出问题时的反应真快"），
差的处理会让一个小 bug 变成丢客户的导火索。

## 何时触发

用户提及以下内容时触发：
- "客户威胁说要取消合同"
- "生产环境挂了，大客户很生气"
- "客户发了 escalation 邮件，抄送了我们的 CEO"
- "怎么处理客户升级？"
- "Customer is threatening to churn over a bug"
- "P0 issue with key account"

**排除：** 此技能 NOT for：
- 客户流失挽回策略 → 用 `churn-prevention`
- 内部团队冲突 → 用 `employee-relations`
- 合同/法律问题 → 咨询法务

## 升级等级

| 等级 | 定义 | 示例 | 响应时间 | 升级到 |
|------|------|------|---------|--------|
| **L1** | 轻微不满 | 单个功能不好用，有 workaround | 24h | CSM |
| **L2** | 影响业务 | 核心功能受阻，影响客户日常运营 | 4h | CSM + 产品 |
| **L3** | 严重故障 | 服务中断、数据丢失风险、安全事件 | 1h | CSM + 工程 + VP |
| **L4** | 关系危机 | 客户威胁取消、法律威胁、社交媒体曝光 | 立即 | CSM + VP + CEO + 法务 |

## 升级处理框架

### 阶段 1：紧急响应（0-2 小时）

**行动清单：**
```
□ 确认事件等级（L1-L4）
□ 通知内部升级链（按等级通知对应层级）
□ 向客户发送首次响应："我们收到了，正在调查"（15 分钟内）
□ 建立 war room（L3+）/ 沟通群
□ 指定单一客户联系人（避免客户收到多个人不同版本的消息）
```

**首次响应话术（15 分钟内发出）：**
```
"[客户联系人]，

我们已收到关于 [问题简述] 的报告，正在紧急调查根因。
我们的技术团队已经开始排查，预计 [X 分钟/小时] 内会有初步结论。

接下来我会每 [X 分钟/小时] 更新一次进展。
如果有紧急问题，我的电话是 [号码]。

[签名]"
```

### 阶段 2：内部调查（2-24 小时）

**行动清单：**
```
□ 根因分析（技术/流程/人为）
□ 评估影响范围（受影响用户数、业务影响程度）
□ 制定修复方案 + ETA
□ 准备客户沟通稿（内部 review 后发送）
```

### 阶段 3：客户沟通（持续）

**沟通节奏：**

| 等级 | 更新频率 | 沟通人 |
|------|---------|--------|
| L1 | 每 24h | CSM |
| L2 | 每 4h | CSM |
| L3 | 每 1h | CSM + VP |
| L4 | 每 30 min | CEO / VP + CSM |

**沟通模板：**
```
"[进度更新 #X] — [日期 时间]

当前状态：[调查中 / 已定位 / 修复中 / 已修复待验证]
根因：[简述]
影响范围：[简述]
预计修复时间：[ETA]
临时 workaround：[如有]

下一步：[我们会做什么 + 下次更新是什么时候]
"
```

### 阶段 4：问题修复

**行动清单：**
```
□ 部署修复
□ 验证修复有效性（不只是开发说"修好了"）
□ 通知客户："已修复，以下是验证结果"
□ 确认客户确认："你那边现在正常了吗？"
```

### 阶段 5：事后复盘

**5 Whys 分析：** 至少问 5 次"为什么"直到找到系统性根因（不是"因为有人写了个 bug"）。

**复盘报告模板：**
```markdown
# 升级事后复盘 — [事件名称] | [日期]

## 事件时间线
| 时间 | 事件 |
|------|------|
| [时间] | 客户首次报告 |
| [时间] | 确认问题 + 升级到 L[X] |
| [时间] | 定位根因 |
| [时间] | 修复部署 |
| [时间] | 客户确认解决 |

## 根因
[技术根因 + 流程根因]

## 改进措施
| # | 措施 | 负责人 | 截止日期 |
|---|------|--------|---------|
| 1 | [防止复发] | [人] | [日期] |
| 2 | [缩短 MTTR] | [人] | [日期] |
| 3 | [改进监控] | [人] | [日期] |

## 客户沟通回顾
- [什么做得好 / 什么需要改进]
```

## L4 危机特别处理

当客户威胁取消、发社交媒体、抄送 CEO 时：

**不要做的事：**
- ❌ 和客户争辩"这不是我们的问题"
- ❌ 承诺你做不到的事（"明天一定修好"）
- ❌ 在内部群里吐槽客户（截图永远会泄露）
- ❌ 让客户自己去找不同部门的人

**要做的事：**
- ✅ 立即通知 VP + CEO
- ✅ 派一个 senior leader 打电话给客户的 senior leader
- ✅ 给一个"我们承认问题 + 正在全力解决 + 具体行动计划"的回复
- ✅ 所有书面沟通假设会被截屏发到社交媒体

## 输出模板

```markdown
# [客户名称] — 升级处理计划

**事件等级：** L[X]  
**事件时间：** [日期 时间]  
**当前状态：** [调查中/修复中/已解决]  

## 影响评估
- 受影响用户数：[X]
- 业务影响：[描述]
- 预计修复时间：[时间]

## 内部对齐
- War room：[链接/群]
- 负责人：[技术] + [CSM] + [管理层]
- 沟通节奏：每 [X] 更新

## 客户沟通计划
- 联系人：[姓名]
- 首次响应：已完成 / 待发送
- 下次更新：[时间]

## 根因与修复
- 根因：[简述]
- 修复方案：[简述]
- ETA：[时间]

## 复盘计划
- 复盘会议：[日期]
- 负责人：[姓名]
```

## 质量原则

1. **速度 > 完美** — 15 分钟内告诉客户"我们知道了"比 2 小时后给完美分析更重要
2. **一个人对外沟通** — 客户只需要一个联系人，内部协调不是客户的事
3. **说话算话** — 说好每小时更新就必须每小时更新，哪怕没进展也要说"仍在调查"
4. **复盘找系统问题** — 不是为了找人背锅，是为了下次不发生
5. **L4 立即到最高层** — 关系危机不能靠 CSM 一个人扛，必须拉上 CEO/VP
