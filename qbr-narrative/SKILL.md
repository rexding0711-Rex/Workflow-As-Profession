# QBR Narrative — 季度业务回顾叙事

好的 QBR 不是"你看我们帮你们做了多少事"，而是"我们一起达成了什么，
接下来一起去哪"。把 QBR 从客户成功团队的"义务劳动"变成客户期待的"战略对话"。

## 何时触发

用户提及以下内容时触发：
- "下周要给客户做 QBR，帮我准备一下"
- "QBR 怎么做才能不让客户觉得是浪费时间？"
- "客户高管要来听季度汇报，怎么讲？"
- "帮我设计 QBR 的结构"
- "Prepare a quarterly business review deck"

**排除：** 此技能 NOT for：
- 客户健康度评估 → 用 `customer-health-score`
- 续约谈判 → 用 `renewal-strategy`
- 客户投诉处理 → 用 `escalation-playbook`

## 开始之前

| # | 信息 | 示例 |
|---|------|------|
| 1 | **客户背景** | "使用我们产品 2 年，年合同 50 万" |
| 2 | **参会人** | "客户方：VP of Engineering + 2 个 Director" |
| 3 | **回顾周期** | "过去 6 个月" |
| 4 | **当前关系阶段** | "稳定使用中，有扩展潜力" |
| 5 | **本次目标** | "推动扩展到另一个部门" |

## QBR 叙事框架

### 黄金结构（45 分钟版）

```
1. 执行摘要 (5 min) — "过去半年我们一起达成了什么"
   └─ 3 个关键数字，让高管在 30 秒内理解价值

2. 价值回顾 (10 min) — "用数据证明 ROI"
   └─ 使用数据 + 业务成果 + 客户证言

3. 洞察与建议 (10 min) — "我们发现了一些你可能没注意到的"
   └─ 行业对标 + 使用差距 + 改进机会

4. 路线图对齐 (10 min) — "我们接下来要去哪"
   └─ 产品路线图 + 客户需求匹配 + 联合计划

5. 下一步与行动 (5 min) — "所以我们建议..."
   └─ 明确的 ask + 双方承诺 + 时间线

6. 开放讨论 (5 min) — 留给客户提问和反馈
```

### 调整版：30 分钟（适合月度 check-in）
- 执行摘要 (3 min) + 价值回顾 (8 min) + 洞察 (8 min) + 下一步 (3 min) + 讨论 (8 min)

### 调整版：60 分钟（适合年度/高管到场）
- 所有模块各加 3-5 min + 增加"行业趋势"和"联合创新"模块

## 每页 slide 的设计公式

### "So What" 原则

每张 slide 包含三层信息：

```
[标题 = 核心结论]
[数据 = 支撑证据]
[So What = 对客户意味着什么]
```

**❌ 错误示例：**
> "过去 6 个月团队登录次数 12,000 次"（数据 dump，没结论）

**✅ 正确示例：**
> "工程师每天在你的平台上花 2.3 小时，比 6 个月前增长 40%"
> → "你们的 CI/CD 流程已经深度依赖我们，我们也看到了更多自动化机会"

### 数据可视化速查

| 要表达 | 图表 | 例子 |
|--------|------|------|
| 增长趋势 | 折线图 | 月活用户数变化 |
| 使用分布 | 热力图 | 各团队功能使用频率 |
| 前后对比 | 配对条形图 | 使用我们前 vs 使用我们后 |
| 达成 vs 目标 | 仪表盘 | 原定 KPI vs 实际达成 |
| ROI | 大数字 | "节省 1,200 小时/年 = ¥600K" |

## 常见场景叙事

### 场景 A：稳定客户 → 推动扩展
**叙事线：** "你们用得很好 → 你们还没用到 XX 功能 → XX 功能能带来 YY 价值 → 我们帮你们启动"
**关键页：** 使用差距分析、竞品对标、扩展 ROI 计算

### 场景 B：使用量下降 → 干预
**叙事线：** "我们注意到使用量变化 → 可能的根因 → 我们的建议 → 一起改进"
**关键页：** 使用趋势图、流失用户分析、干预方案

**注意：** 不要回避问题。客户早就知道使用在下降，假装没看到更糟。诚实 + 解决方案 = 信任。

### 场景 C：新客户 → 建立信任
**叙事线：** "上线以来我们完成了什么 → 行业 benchmark 对比 → 接下来 90 天路线图"
**关键页：** Quick wins 列表、onboarding 完成情况、行业对标

## 高管在场时的特别调整

当客户方有 VP 或 C-level 参会：

1. **第一页就要给结论** — "过去 6 个月，我们在 3 个关键指标上达成了 XX"
2. **去掉操作细节** — 不讲"你们有多少个工单已解决"，讲"我们的响应速度改善了多少"
3. **用他们的语言** — 讲 revenue impact、risk reduction、time savings，不讲功能列表
4. **准备一个 bold ask** — 高管来了就不能只谈日常工作，要有一个推进关系的请求

```markdown

**会议日期：** [日期]  
**参会人：** [我方] + [客户方]  
**时长：** [30/45/60] 分钟  

---

## Slide 1: 执行摘要
**核心信息：** [一句话总结最大价值]
**3 个关键数字：**
1. [数字 1 + 含义]
2. [数字 2 + 含义]
3. [数字 3 + 含义]

## Slide 2-4: 价值回顾
[按"使用数据 → 业务成果 → So What"展开]

## Slide 5-6: 洞察与建议
[发现 + 建议 + 预期价值]

## Slide 7: 路线图对齐
[我们的方向 + 对客户的影响 + 联合计划]

## Slide 8: 下一步
**我们的 Ask：** [明确的请求]
**双方承诺：** [我们做什么 / 客户做什么]
**时间线：** [日期]
```

## 质量原则

1. **一页一个观点** — 不要在一张 slide 里塞 5 个信息
2. **数据必须有自己的"so what"** — 没有解读的数据 = 浪费客户的注意力
3. **诚实比完美重要** — 使用量下降了就说下降了，加一句"我们正在解决"
4. **Bold Ask 必须有** — 每次都推动关系前进一步（case study / reference / expansion / introduction）
5. **提前发 deck，会上只讲重点** — QBR 不是阅读课，是对话

## 使用示例

**用户输入：**
> 一个使用我们一年的 SaaS 客户，下周三 QBR。他们 CTO 要来。合同快到期了，想推动升级到企业版。

**AI 执行过程：**
1. 判断为场景 A（稳定客户→推动扩展）+ 高管在场调整
2. 收集使用数据和业务成果
3. 按黄金结构生成叙事线：价值回顾 → 使用差距（企业版功能） → ROI 计算 → Ask
4. 重点设计"为什么升级到企业版"的 story（不是功能列表，是商业价值）
5. 输出 slide-by-slide 内容 + 现场话术建议
