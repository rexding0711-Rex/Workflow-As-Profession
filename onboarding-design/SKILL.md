# Onboarding Designer — 新人入职方案设计

设计从签约到胜任的完整入职体验。入职不是填表 + 发电脑，而是让新人在最短时间内从"我是谁？我该做什么？"变成"我知道我要做什么，我有人可以问，我看到我的价值"。

## 何时触发

用户提及以下内容时触发：
- "帮我设计一个新人入职流程"
- "新员工来了一个月就走了，怎么改进 onboarding？"
- "要招 10 个应届生，怎么做入职培训？"
- "Design a 30-60-90 day onboarding plan"
- "怎么配 buddy？"
- "入职体验很差，想重构"

**排除：** 此技能 NOT for：
- 招聘流程 → 用 `jd-writer`、`structured-interview`
- 绩效评估 → 用 `performance-review`

## 开始之前

### 必收集项

| # | 信息 | 示例 |
|---|------|------|
| 1 | **岗位类型** | "技术（后端）/ 销售 / 管理" |
| 2 | **团队规模** | "现有 30 人，每月入职 2-3 人" |
| 3 | **新人背景** | "社招为主" / "校招为主" / "混合" |
| 4 | **当前痛点** | "没人带" / "不知道做什么" / "流程乱" |

### 加分收集项

| # | 信息 | 示例 |
|---|------|------|
| 5 | 公司文化特点 | "扁平、文档驱动、远程优先" |
| 6 | 现有 onboarding 有什么 | "只有 HR 入职手续，没有岗位培训" |
| 7 | 工具栈 | "飞书 + Notion + GitHub" |

## 工作模式

### 模式 A：从零搭建
为没有正式 onboarding 的团队设计完整方案。Pre-boarding → Day 1 → Week 1 → Month 1-3。

### 模式 B：现有流程诊断
分析现有 onboarding 的问题，给出改进方案和优先级排序。

### 模式 C：单岗位入职路径
为特定岗位（如"后端工程师"或"销售经理"）设计定制化的入职路径。

## 入职周期框架

### 阶段 0：Pre-boarding（签约 → 入职前）

| 时间 | 行动 | 负责人 |
|------|------|--------|
| 签约后 24h | 发送 Welcome Email（含入职日期、准备事项） | HR |
| 入职前 2 周 | 设备准备（电脑、账号、权限） | IT |
| 入职前 1 周 | 指定 Buddy + 发送 Buddy 指南 | HR + Buddy |
| 入职前 3 天 | 发送 Day 1 日程 | HR |

**Welcome Email 模板：**
```
Hi [姓名]，

恭喜加入 [公司]！我们期待 [入职日期] 与你见面。

在你入职前，这里有 3 件事：
1. 你的 Buddy 是 [姓名+岗位]，ta 会主动联系你
2. 入职第一天安排见附件
3. 有任何问题随时找我 [联系方式]

[公司] HR Team
```

### 阶段 1：Day 1 — 归属感

| 时间 | 内容 | 负责人 |
|------|------|--------|
| 9:00 | 工位准备 + 设备就绪检查 | IT + Buddy |
| 9:30 | 团队欢迎（咖啡/早餐 informal） | Buddy |
| 10:00 | HR 入职手续（30 min，不超过） | HR |
| 10:30 | Manager 1-on-1：角色使命 + 30 天期望 | Manager |
| 13:00 | 公司愿景/产品/文化介绍 | 创始人/Leader |
| 14:00 | 工具和系统设置（权限、账号、IDE） | IT + Buddy |
| 16:00 | 第一个小任务布置（今天就能完成的 win） | Manager |

> Day 1 核心目标：**新人回家时觉得"今天没白来，我知道我为什么在这了"**

### 阶段 2：Week 1 — 基础建立

| 任务 | 目标 |
|------|------|
| 完成开发环境/工作环境搭建 | 能独立工作 |
| 参加 ≥ 3 个团队会议（观察为主） | 理解团队协作方式 |
| 阅读核心文档（架构/产品/流程） | 建立上下文 |
| 与每个团队成员 15 min 1-on-1 | 认识人、知道找谁 |
| 完成第一个小任务 | 体验完整工作流 |
| 提交 PR / 文档 / 交付物 | 建立成就感 |

### 阶段 3：Month 1 — 独立贡献

**30 天检查清单：**
- [ ] 能独立完成常规任务
- [ ] 了解团队目标和自己负责的部分
- [ ] 知道遇到 X 问题应该找 Y 人
- [ ] 至少参与 1 个完整项目/功能交付
- [ ] Manager 完成第一次 30 天回顾

**30 天回顾（Manager 1-on-1）：**
```
1. 这 30 天，最有成就感的是什么？
2. 什么让你最困惑？
3. 入职前 30 天和你的预期有什么不同？
4. 下 30 天你最想深入的是什么？
```

### 阶段 4：Month 2-3 — 深度融入

**60 天检查点：**
- [ ] 开始主导 1 个小项目或独立模块
- [ ] 参加跨团队会议/项目
- [ ] 至少提出 1 个改进建议

**90 天检查点：**
- [ ] 完成试用期评估（与 `performance-review` 衔接）
- [ ] 360 度收集同事反馈（2-3 人）
- [ ] Manager 和新人共同确认：岗位匹配度、成长方向

## Buddy 机制设计

### Buddy 选择标准

| 标准 | 说明 |
|------|------|
| 同岗位/相关岗位 | 能回答技术问题 |
| 入职 ≥ 6 个月 | 熟悉公司流程和文化 |
| 非直属上级 | 新人更放松交流 |
| 自愿 | 强制分配的 buddy 效果打折扣 |

### Buddy 职责

| 时间 | 职责 |
|------|------|
| 入职前 | 发欢迎消息，回答入职前问题 |
| Day 1 | 带吃午饭、办公室 tour |
| Week 1 | 每天 5 min check-in |
| Month 1 | 每周 1 次 15 min 1-on-1 |
| Month 2-3 | 随缘，但确保新人知道随时可以找 |

## 岗位特定培训路径

### 技术岗位（工程师）

```
Week 1: 环境搭建 + 代码库导览 + 第一个 bug fix
Week 2-3: 结对编程 + 小功能开发
Month 1: 独立负责一个模块
Month 2-3: 参与架构评审 + on-call 轮值
```

### 销售岗位

```
Week 1: 产品深度学习 + 客户画像熟悉
Week 2-3: 旁听资深销售的客户会议 + 模拟 pitch
Month 1: 独立跟进 5-10 个小客户
Month 2-3: 主导一个完整的销售周期
```

### 管理岗位

```
Week 1-2: 与所有直接下属 + 跨部门负责人 1-on-1
Week 3-4: 旁听现有管理会议 + 了解决策机制
Month 1: 制定团队 90 天计划
Month 2-3: 独立主持会议 + 做出 2-3 个管理决策
```

## 入职体验反馈

新人入职第 14 天、30 天、90 天各发一次简短问卷：

```
1. 入职体验整体评分（1-10）：__
2. 入职前和入职后的落差最大的地方：__
3. 目前最大的困难：__
4. Buddy/Manager 的支持够不够？__
5. 有什么建议让下一个新人体验更好？__
```

## 质量原则

1. **Day 1 一定要给一个小任务** — 当天完成、有可见产出。入职第一天逛 wiki 是最差体验
2. **Buddy 不是新人保姆** — 是向导，"带你看地图"不是"替你走路"
3. **Pre-boarding 别忽略** — 签约到入职的 2-4 周是"反悔高发期"，保持温度
4. **入职不是填表** — 减少 HR 行政环节占总时长的比例（目标 ≤ 20%）
5. **反馈要闭环** — 收了问卷不跟进 = 比不发问卷还糟糕

## 使用示例

**用户输入：**
> 我们是一个 30 人 SaaS 公司，目前每月入职 2-3 个工程师。现在 onboarding 就是 HR 发台电脑然后全靠自学，新人都说没人带。帮我设计一个方案。

**AI 执行过程：**
1. 判断为模式 A：从零搭建（无正式 onboarding）
2. 追问公司文化和工具栈
3. 输出完整方案：Pre-boarding → Buddy 选择 → Day 1 日程 → 30-60-90 路线图
4. 重点设计 Buddy 机制（当前核心痛点是"没人带"）
5. 附反馈问卷模板
