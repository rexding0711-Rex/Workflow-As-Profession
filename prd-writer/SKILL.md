# PRD Writer — 产品需求文档撰写

差的 PRD：开发看着看着开始骂 PM。好的 PRD：开发看着看着开始写代码。
区别不在于文笔，在于有没有回答开发真正想问的 5 个问题：
"为什么要做？给谁用？怎么算做完？边界在哪？什么情况算异常？"

## 何时触发

用户提及以下内容时触发：
- "帮我写一个 PRD"
- "这个需求帮我整理成文档"
- "Write a product requirements document"
- "给开发的需求说明怎么写？"
- "老板给了个想法，怎么落地成具体需求？"

**排除：** 此技能 NOT for：
- 产品战略和路线图 → 用 `product-strategy`
- 用户故事拆分和 backlog → 用 `user-story-mapping`
- 跨部门 stakeholder 管理 → 用 `stakeholder-alignment`

## 开始之前

| # | 信息 | 示例 |
|---|------|------|
| 1 | **要解决什么问题** | "用户注册流程流失率 40%" |
| 2 | **目标用户** | "中小企业主，非技术背景" |
| 3 | **业务目标** | "3 个月内注册转化率提升到 70%" |
| 4 | **约束条件** | "不能动后端 API（这赛季的 OKR 不在那）" |

## PRD 标准结构

### 8 段式 PRD

```
1. 背景与问题 — 不写这一章，后面都是自说自话
2. 目标与成功指标 — 怎么衡量"做完了"和"做得好"  
3. 用户场景 — 谁在什么情况下怎么用
4. 功能范围 — 做哪些、不做哪些（MVP vs 后续迭代）
5. 功能详述 — 每个功能的交互逻辑和边界条件
6. 非功能需求 — 性能、安全、兼容性、埋点
7. 依赖与风险 — 需要谁配合、可能出什么错
8. 发布计划 — 分几期、每期验收什么
```

## 各段写法指南

### 1. 背景与问题
**回答：** "为什么现在要花时间做这个？"
- 用数据说话：不是"用户觉得注册太麻烦"，是"从注册第一步到完成的流失率 40%，行业基准 25%"
- 如果来自用户反馈：引用具体用户的原话
- 一句话说明不做会怎样

### 2. 目标与成功指标
**区分"做完"和"做得好"：**

| 指标 | 当前值 | 目标值 | 测量方式 |
|------|--------|--------|---------|
| 注册转化率 | 60% | 70% | 漏斗分析 |
| 注册时长 | 4.5 min | <2 min | 前端埋点 |
| 注册后的激活率 | 35% | 50% | 7 天内完成核心动作 |

### 3. 用户场景
**用 Job Story 格式替代传统 User Story：**
```
当 [情况/上下文]，我想 [动机/驱动力]，以便 [期望结果]。
```

**❌ 传统 User Story：** "作为一个用户，我想快速注册，以便使用产品。"（废话）
**✅ Job Story：** "当我第一次听说这个产品并想试用时，我想用手机号快速开始，以便在 30 秒内看到产品核心价值，而不是填 10 个表单。"

### 4. 功能范围（In/Out 表）

| 范围 | 内容 |
|------|------|
| ✅ MVP（本期） | [列核心功能] |
| ❌ Out of Scope | [明确不做，避免后期需求蔓延] |
| 🔮 下期考虑 | [如果 XX 数据达到 YY，考虑做 ZZ] |

### 5. 功能详述
**每个功能点包含：**
```
### [功能名称]

**交互描述：** [用户做了什么、系统如何响应]
**边界条件：** [空数据、最大值、网络异常、权限不足时怎么办]
**验收标准：**
- [ ] [可测试的条件 1]
- [ ] [可测试的条件 2]
- [ ] [可测试的条件 3]
```

### 6. 验收标准写法

**Gherkin 格式（Given-When-Then）：**
```
Feature: 手机号快速注册
  Scenario: 新用户用手机号注册
    Given 用户在注册页
    When 输入手机号 "13800138000" 并点击"获取验证码"
    Then 系统发送验证码到该手机号
    And 验证码输入框获得焦点
    And 60 秒内"获取验证码"按钮置灰不可点击
    
  Scenario: 手机号已被注册
    Given 用户在注册页
    When 输入已注册手机号 "13800138001"
    Then 系统提示"该手机号已注册，请直接登录"
    And 显示"去登录"链接
```

## 质量原则

1. **先写 Why 再写 What** — 跳过背景直接写功能 = 开发不知道为什么做，做出奇怪的东西
2. **验收标准必须可测试** — "用户体验好"不是 AC，"注册流程 ≤3 步且 ≤90 秒"才是
3. **明确 In/Out 边界** — PRD 最大的敌人是"顺便做一下"
4. **边界条件 ≥3 个** — 空状态、极限值、异常情况。只写 happy path 的 PRD 是半成品
5. **PRD 不是写给自己看的** — 一个新加入的工程师读一遍就能开干，才是合格的

## 使用示例

**用户输入：**
> 老板说要在电商 App 里加一个"帮我选"功能，AI 根据用户的浏览和购买记录推荐商品。帮我写个 PRD。

**AI 执行过程：**
1. 追问：目标用户？要提升什么指标？约束条件？
2. 按 8 段式结构展开，重点打磨"AI 推荐的解释逻辑"（Why this recommendation）的验收标准
3. 明确 Out of Scope：第一个版本不做实时对话推荐（需要 NLP 团队配合）
4. 非功能需求：推荐响应时间 <500ms、埋点跟踪推荐点击率
