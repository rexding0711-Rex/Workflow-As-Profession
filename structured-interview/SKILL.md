# Structured Interview Designer — 结构化面试设计

为面试官设计基于能力的结构化面试。
研究证明非结构化面试的预测效度仅 0.2，而结构化面试可达 0.4-0.6。本技能帮你把面试从"聊天"升级为"科学评估"。

## 何时触发

用户提及以下内容时触发：
- "帮这个岗位设计面试题"
- "我要面一个高级产品经理，问什么好？"
- "帮我做一个面试评分表"
- "怎么判断候选人的系统设计能力？"
- "Design interview questions for engineering manager"

**排除：** 此技能 NOT for：
- 候选人模拟面试练习 → 用 `interview-simulator`
- JD 撰写 → 用 `jd-writer`

| 3 | **面试时长** | "45 分钟" |
| 4 | **面试轮次定位** | "技术二面" 或 "终面" |

| # | 信息 | 示例 |
|---|------|------|

| 6 | 面试官经验水平 | "新手面试官，需要详细追问指南" |
| 7 | 候选人背景（如有） | "5 年大厂经验" |

## 工作模式

### 模式 A：全流程设计
从能力模型 → 面试题 → 评分 rubric → 面试官指南，完整的面试设计包。

### 模式 B：单场景深挖
用户只关注某个维度（如只设计系统设计题），输出针对该维度的深度题库 + 评估标准。

### 模式 C：Rubric 优化
用户已有面试题，但缺少评分标准。为每道题设计 1-5 分行为锚定评分表（BARS）。

## 能力模型提取

从 JD 中提取 3-5 个核心能力维度，每个对应一种面试题型：

| 能力维度 | 面试方法 | 权重 |
|---------|---------|------|
| 技术硬技能 | 技术问答 / Pair Programming | 30-40% |
| 问题解决 | 案例题（Case Study） | 20-30% |
| 系统设计 / 架构 | 设计题（Design Task） | 20-30% |
| 沟通协作 | BEI 行为事件访谈 | 10-15% |
| 领导力 / 影响力 | BEI + 情境题（Situational） | 10-15% |

> 权重根据岗位性质调整。IC（独立贡献者）岗位：技术硬技能 40% + 问题解决 30%；管理岗：领导力 30% + 沟通协作 25%。

## 面试题库

### STAR 行为面试题标准格式

```markdown
**【题目】**
**考察能力：** [能力维度]
**题面：** "请分享一个你 [具体情境] 的经历。当时你面临什么挑战？你采取了什么行动？结果如何？"
**追问指南：**
- [追问 1：深挖行动细节]
- [追问 2：验证结果真实性]
- [追问 3：检验迁移能力"如果重来一次你会怎么做？"]
```

### BEI 行为事件访谈题

```
情境题（Situational）：
"假设你加入了团队，发现代码 review 流程形同虚设，你会怎么推动改进？"

过去行为题（Past Behavioral）：
"请分享一个你曾经推动过的最有挑战的技术决策，从提案到落地的全过程。"
```

## 评分 Rubric 设计

每道面试题输出 5 级行为锚定评分：

| 分数 | 等级 | 行为锚定示例 |
|------|------|------------|
| **5** | 卓越 | 不仅回答了方案，还主动分析了 trade-off、考虑了边界条件、给出了可落地的实施路线图 |
| **4** | 优秀 | 方案完整、逻辑清晰、能回答大部分追问 |
| **3** | 合格 | 方案基本正确，但在深度追问下暴露出盲区 |
| **2** | 待提升 | 方案有明显漏洞，或需要大量提示才能完成 |
| **1** | 不通过 | 无法给出有效方案，或方案与岗位要求严重不匹配 |

## 面试官指南

### 追问技巧

1. **漏斗追问法**：开放式 → 具体化 → 验证
   - "能分享一个处理线上事故的例子吗？" → "当时你是怎么定位根因的？" → "事故复盘后你推动了什么流程改进？"
2. **STAR 完整性检查**：候选人是否漏了 Situation / Task / Action / Result 任一环节
3. **虚构检测**：如果候选人的回答缺少具体细节（人名、数据、时间），追问"当时有几个团队成员参与？"

### 常见偏见提醒

| 偏见类型 | 表现 | 避免方法 |
|---------|------|---------|
| 确认偏误 | 前 5 分钟形成印象，后面只找支持证据 | 每题独立打分，最后汇总 |
| 相似偏误 | 喜欢和自己背景像的人 | 关注"能力差异"而非"文化 fit" |
| 光环效应 | 因为一个亮点忽略其他短板 | 每个维度独立评分 |
| 对比效应 | 受前一个候选人影响 | 使用行为锚定标准而非相对判断 |

详见 `references/bias-checklist.md`

## 质量原则

1. **每题考察单一能力维度** — 一题测多维 = 什么都测不准
2. **追问 ≥ 2 条** — 结构化的核心是追问，不是念题
3. **评分标准必须可观察** — 不是"态度好"，而是"被追问时能保持冷静并给出具体回应"
4. **题库有冗余** — 设计 6-8 题，实际使用 3-4 题，留备用
5. **所有题目必须关联岗位 JD** — 不考与岗位无关的能力

## 与其他技能的关系

- 面试方案生成后 → 可用 `jd-writer` 的能力对 JD 做交叉检查
- 面试数据积累后 → 可用 `people-analytics` 分析面试通过率、评估效度
- 本技能 ≠ `interview-simulator`（候选人练习用）

## 使用示例

**用户输入：**
> 帮我设计一套后端工程师技术二面的面试题，侧重系统设计和问题排查能力，45 分钟。

**AI 执行过程：**
1. 追问：级别、当前团队、面试官经验
2. 提取能力模型：系统设计(40%) + 问题排查(35%) + 沟通(25%)
3. 生成 6 道 STAR/BEI 题，含完整追问链
4. 输出 5 级行为锚定评分 rubric
5. 附面试官注意事项（时间分配、红色信号）
