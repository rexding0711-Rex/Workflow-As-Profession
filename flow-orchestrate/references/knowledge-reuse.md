# 知识复用机制

## 知识资产定义

项目中产生的可复用知识：

| 知识类型 | 来源 | 复用价值 | 存储位置 |
|----------|------|----------|----------|
| 行业认知 | 调研项目 | 新项目快速启动 | knowledge-base/industry/ |
| 分析方法 | 分析项目 | 同类分析直接套用 | knowledge-base/methods/ |
| 决策模板 | 决策项目 | 评分矩阵/权重参考 | knowledge-base/decisions/ |
| 内容模板 | 内容项目 | 文档框架复用 | knowledge-base/templates/ |
| 执行清单 | 执行项目 | 任务清单/SOP复用 | knowledge-base/checklists/ |
| 供应商库 | 评估项目 | 供应商信息复用 | knowledge-base/vendors/ |
| 风险清单 | 各类项目 | 常见风险识别 | knowledge-base/risks/ |
| 复盘经验 | 复盘项目 | 教训/最佳实践 | knowledge-base/lessons/ |
| 人脉网络 | 各类项目 | 关键联系人 | knowledge-base/contacts/ |
| 数据资产 | 调研项目 | 基础数据/基准值 | knowledge-base/data/ |

## 相似项目检索

新项目启动时，自动检索相关经验：

```
检索维度：
1. 项目类型匹配（新赛道调研→新赛道调研经验）
2. 关键词匹配（陶瓷→散热→半导体→材料）
3. 客户/行业匹配（数据中心→光模块→功率半导体）
4. 时间衰减（近期经验优先，但不过滤过期经验）

检索结果：
- 高相关度（>80%）：直接引用，输出"基于xx项目经验，建议..."
- 中相关度（50-80%）：参考借鉴，输出"xx项目有类似情况，当时..."
- 低相关度（<50%）：归档备查，不主动引用
```

## 经验引用机制

### 在项目启动时引用

```markdown
# 项目启动建议

基于历史相似项目经验，以下认知可直接复用：

## 直接复用（高置信度）
- [EXP-2024-001] 陶瓷基板散热市场处于渗透拐点，AI算力是核心驱动力
  → 本项目可直接引用，无需重新验证
- [EXP-2024-001] 日本厂商占60%份额，国内追赶中
  → 竞争格局认知可直接引用，建议更新数据

## 参考借鉴（需验证）
- [EXP-2024-002] 新材料调研中，供应商会夸大能力
  → 建议本项目访谈时保持警惕，交叉验证
- [EXP-2024-003] 客户访谈需提前准备案例公司
  → 建议复用该方法，但需适配新客户群体

## 推荐方法/模板
- 推荐模板：[EXP-2024-001] 陶瓷散热调研报告模板
- 推荐方法：[EXP-2024-001] 散热技术调研框架
- 推荐提纲：[EXP-2024-001] 散热需求访谈提纲

## 推荐联系人
- 技术专家：xxx（陶瓷基板工艺，已验证可靠性高）
- 客户：xxx公司（光模块散热需求，可能愿意二次访谈）
```

### 在执行过程中引用

```markdown
# 执行提醒

当前任务「供应商访谈」与历史经验关联：

- [EXP-2024-001] 供应商夸大能力风险
  → 建议访谈时增加：客户案例验证、第三方报告交叉、现场考察
- [EXP-2024-001] 访谈提纲可直接复用
  → 已自动加载「散热需求访谈提纲」
```

### 在决策时引用

```markdown
# 决策参考

类似场景历史决策：

- [EXP-2024-001] 技术选型：当时选择了方案B，原因是...
  → 当前场景与历史相似度70%，建议参考但不照搬
- [EXP-2024-002] 投资决策：当时NPV门槛设为200万，实际...
  → 建议本项目NPV门槛调整至150万
```

## 知识更新机制

### 经验卡片生命周期

```
创建 → 引用 → 验证 → 更新 → 归档

- 创建：项目复盘后自动生成
- 引用：新项目启动时检索
- 验证：引用时标注置信度，执行后验证是否仍有效
- 更新：季度回顾，更新数据/认知/联系人
- 归档：过期（2年无引用）或失效（认知被颠覆）时归档
```

### 置信度衰减规则

| 经验类型 | 初始置信度 | 衰减周期 | 衰减后处理 |
|----------|------------|----------|------------|
| 市场认知 | 高/中/低 | 6个月 | 需重新验证 |
| 技术认知 | 高/中/低 | 12个月 | 需重新验证 |
| 竞争格局 | 高/中/低 | 3个月 | 需更新数据 |
| 分析方法 | 高 | 长期有效 | 持续复用 |
| 模板 | 中 | 每次复用优化 | 版本迭代 |
| 人脉 | 高/中 | 每次使用前确认 | 更新联系方式 |
| 数据基准 | 中 | 6个月 | 更新数据 |

### 经验验证机制

```
每次引用经验后，执行验证：
1. 该经验是否仍然有效？（有效/部分有效/失效）
2. 如果有偏差，偏差是什么？
3. 是否需要更新经验卡片？
4. 验证结果自动反馈到经验卡片，更新置信度
```

## 知识复用度量

### 复用指标

| 指标 | 计算方式 | 目标 |
|------|----------|------|
| 经验引用率 | 引用经验的项目数/总项目数 | >60% |
| 方法复用率 | 复用方法的项目数/总项目数 | >50% |
| 模板复用率 | 复用模板的项目数/总项目数 | >70% |
| 数据复用率 | 复用数据的项目数/总项目数 | >40% |
| 新人上手时间 | 有知识库 vs 无知识库 | 缩短50% |
| 重复调研率 | 重复调研同一主题的比率 | <20% |

## 与其他技能的关系

```
review-all → 知识库 → flow-orchestrate / monitor / analyze / decide / create / execute
    ↓              ↓                    ↓
  经验输出      存储管理              检索引用
```

- review-all 输出经验卡片到知识库
- flow-orchestrate 启动时自动检索相关经验
- 各层技能在执行过程中可引用相关经验
- 知识库定期维护，更新置信度，清理过期经验
