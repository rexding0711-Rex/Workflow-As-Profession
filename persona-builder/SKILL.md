# Persona Builder — 用户画像与旅程图

一个好的 Persona 能让团队在争论"用户会不会喜欢这个功能"时，
不再拍脑袋——而是看着同一个 Persona 说："小李（我们的主要用户画像）会用吗？怎么用？"
差劲的 Persona 是"25-35 岁、本科、住一线城市"的人口统计垃圾。
好的 Persona 是"小李每天有 3 小时在仓库里，手机信号时有时无，他最怕的事是扫码扫不上"。

## 何时触发

用户提及以下内容时触发：
- "帮我构建一个用户画像"
- "我们的用户到底是谁？"
- "需要一个用户旅程图"
- "Create a user persona"
- "JTBD 分析怎么做？"
- "用户在我们 App 里的情绪是怎么变化的？"

**排除：** 此技能 NOT for：
- 用户访谈执行 → 用 `user-interview-guide`
- 可用性测试 → 用 `usability-testing`

## Persona 构建

### 基于数据的 Persona 模板

```markdown
# [Persona 名称] — [一句话概括]
*"一个能引用用户原话的句子"*

## 人口统计（只写对行为有影响的）
- 角色：[岗位]
- 行业：[行业 + 公司规模]
- 经验：[与你的产品相关的经验]

## 行为特征（最重要）
- 使用频率：[每天 / 每周 / 偶尔]
- 使用场景：[在哪里、什么时间、用什么设备]
- 核心目标：[用你的产品要完成什么]
- 当前替代方案：[如果没有你的产品，他在用什么]
- 技术能力：[小白 / 能学 / 专家]

## 痛点
1. [最大的痛点 — 用用户原话]
2. [次大痛点]
3. [其他摩擦]

## 动机
- 要完成的核心 JTBD：[当XX情况，我想YY，以便ZZ]
- 深层需求：[表面需求背后的心理需求]

## 决策因素
- 选你的原因：[为什么选择了你的产品]
- 弃用的原因：[什么会让他离开]
```

### Persona 不是营销画像

| 营销画像 | Persona |
|---------|---------|
| "25-35 岁女性，月入 ¥15K" | "每天在通勤地铁上用碎片时间处理工作消息" |
| 基于人口统计 | 基于行为和需求 |
| 用来做广告投放 | 用来做产品设计决策 |

## 用户旅程图

### 五阶段标准结构

```
意识 → 考虑 → 决策 → 使用 → 忠诚/流失
```

### 旅程图模板

| 阶段 | 用户行为 | 触点 | 想法/情绪 | 痛点 | 机会 |
|------|---------|------|---------|------|------|
| 意识 | [用户在做什么] | [接触了哪里] | 😤😐😊 | [痛点] | [你的机会] |
| 考虑 | [同上] | [同上] | ... | ... | ... |

### 情绪曲线

在旅程的每个阶段标注用户的情绪：😤（沮丧）😐（中性）😊（满意）

如果你发现用户在某个节点情绪骤降——那就是你的核心优化机会。

## JTBD 框架

### 核心公式

```
当 [情况/上下文]
我想 [动机/驱动力]
以便 [期望结果]
```

### JTBD 的四个维度

| 维度 | 要问的问题 |
|------|-----------|
| **功能** | 用户要完成什么任务？ |
| **情感** | 用户想感受到什么？（自信？安心？效率？） |
| **社交** | 用户希望别人怎么看他？（"我推荐这个给老板显得我有眼光"） |
| **场景** | 在什么情况下触发？（时间、地点、设备、心情） |

### 从"我买了"到 JTBD 的推导过程

```
用户行为：在 App 上用微信支付买了健身课

❌ 浅层解读："用户喜欢微信支付"（然后你怎么优化？）

✅ JTBD 解读：
   "当我临时决定去健身房但忘带钱包（情况），
   我想快速完成支付以免尴尬地占着柜台（动机），
   以便我像那些老会员一样自然地走进健身房（情感+社交）。"
   
   → 设计启示：支付速度 > 支付选项的丰富度。自动记住上次支付方式。
```

```markdown
| 角色 | [岗位] |

| 最大痛点 | [痛点] |

## 次要 Persona（如有）
[同上，简要版]

## 用户旅程图

| 阶段 | 行为 | 触点 | 情绪 | 痛点 | 机会 |
|------|------|------|------|------|------|
| 意识 | [行为] | [触点] | 😊/😐/😤 | [痛点] | [机会] |

## 关键 JTBD

```

## 质量原则

1. **Persona 必须基于数据而非假设** — 一个没做过用户访谈的 Persona 是小说，不是工具
2. **1-3 个主要 Persona 就够了** — 超过 5 个说明你没想清楚谁才是重点用户
3. **Persona 要有名字、要有照片、要有原话** — 不是"高级用户"，是"小李，仓库管理员，他说'我要是在仓库里扫不上码，后面 10 辆车都得等'"
4. **旅程图里情绪最低的点 = 你的排期优先级** — 先修那些让用户想砸手机的
5. **JTBD 是稳定的，解决方案不是** — "我想在移动中听音乐"是 JTBD，随身听、iPod、Spotify 都是不断变化的解决方案
