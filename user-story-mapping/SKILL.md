# User Story Mapping — 用户故事地图

需求清单（Backlog）最大的问题是：它是一维的。你盯着 200 个 P0/P1/P2 的故事，
分不清哪个先做哪个后做。用户故事地图把一维的 Backlog 变成二维的地图——
横轴是用户旅程，纵轴是优先级。一眼看出 MVP 的边界线。

## 何时触发

用户提及以下内容时触发：
- "帮我把这些需求排个优先级"
- "太多需求了，怎么拆 MVP？"
- "Create a user story map"
- "Backlog grooming 怎么做？"
- "Epic 太大不知道怎么拆"
- "Sprint 0 做什么？"

**排除：** 此技能 NOT for：
- PRD 撰写 → 用 `prd-writer`
- 产品战略和路线图 → 用 `product-strategy`

## 开始之前

| # | 信息 | 示例 |
|---|------|------|
| 1 | **产品/功能名称** | "移动端下单流程优化" |
| 2 | **目标用户** | "C 端消费者，25-40 岁" |
| 3 | **用户旅程主线** | "浏览→加购→结算→支付→查看订单" |
| 4 | **已知的需求清单** | "有一大堆用户反馈和老板需求" |

## 故事地图构建方法

### 五步法

```
Step 1: 画骨架（用户旅程）
  └─ 用户完成一个目标需要经过哪几步？5-10 步，每步是一个"用户活动"
  
Step 2: 填故事（每个步骤下的用户任务）
  └─ 在这个步骤中，用户需要做什么？
  
Step 3: 排优先级（从上到下）
  └─ 第一行 = 必不可少（没有它，用户无法完成基本目标）
  └─ 第二行 = 非常重要（没有它体验很痛苦）
  └─ 第三行 = 有了更好
  
Step 4: 切 MVP 线
  └─ 在第一行上方画一条线，上面的 = MVP
  
Step 5: 切发布切片
  └─ 每个发布版本切一个横向的"可交付增量"
```

### 故事地图模板

```
旅程步骤: [发现] → [浏览] → [决策] → [购买] → [使用] → [分享]
    ↓
第1行：→ 搜索    → 商品详情 → 加购    → 结算    → 查看订单 → 评价
(MVP)  → (基本)  → (基本)   → (基本)  → (基本)  → (基本)   → (基本)

第2行：→ 推荐    → 对比     → 收藏    → 优惠券  → 物流追踪 → 返利
(R2)   → 筛选    → 评价     → 降价提醒 → 分期    → 客服      → 邀请

第3行：→ AR 试穿  → 视频    → 搭配推荐 → 送礼    → 社区     → 达人
(R3)
```

## 需求拆分规则

### Epic → Story 拆分

一个 Epic 很大（"用户下单体验优化"），拆成多个人们可以在一个 Sprint 内完成的 Story。

**拆分维度：**

| 维度 | 示例 |
|------|------|
| 按操作 | "手机号登录"、"微信登录"、"Apple ID 登录" |
| 按角色 | "管理员删除用户"、"用户自注销" |
| 按数据变体 | "文本输入"、"图片上传"、"视频上传" |
| 按状态 | "正常流程"、"网络异常"、"未登录" |

### 拆得对不对？SPLIT 检查

每个 Story 用 SPLIT 法则自检：

| 法则 | 检查 |
|------|------|
| **S**mall | 一个 Sprint 内能交付吗？（≤5 天/人） |
| **P**roducible | 交付后有可见的产出吗？用户能感知吗？ |
| **L**earnable | 做完后团队学到了什么？（不是纯苦力） |
| **I**ndependent | 可以独立开发、测试、部署吗？ |
| **T**estable | 有明确的"做完"标准吗？ |

## 优先级排序

### MoSCoW 方法

| 优先级 | 含义 | 比例建议 |
|--------|------|---------|
| **Must have** | 没有就不叫"完成"，MVP 必须有 | ≤60% |
| **Should have** | 非常重要，但有 workaround 可忍一个版本 | ~20% |
| **Could have** | 锦上添花，资源允许就做 | ~15% |
| **Won't have** | 明确这一版不做 | ~5% |

### 其他评分维度

除了 MoSCoW，加两个维度：

| Story | 用户价值 (1-5) | 实现成本 (1-5) | 价值/成本比 | 优先级 |
|-------|-------------|-------------|-----------|--------|
| 微信登录 | 4 | 2 | 2.0 | P0 |
| 人脸登录 | 2 | 5 | 0.4 | P2 |

> 价值/成本比 >1.5 = 高优先级；<0.5 = 除非老板拿着刀逼你

## 迭代规划

### Release 切片

```
MVP (R1): 用户能完成基本下单（浏览→加购→支付）
  6 周 / 3 Sprints / 15 Stories

Release 2: 提升转化（推荐 + 优惠券 + 评价）
  4 周 / 2 Sprints / 10 Stories

Release 3: 社交化（邀请 + 返利 + 达人社区）
  取决于 R2 发布后的数据
```

### Sprint 容量规划

| Sprint | Story Points | 故事数 | 主题 |
|--------|-------------|--------|------|
| Sprint 1 | 21 | 5 | 商品浏览 + 搜索 |
| Sprint 2 | 21 | 5 | 加购 + 结算 |
| Sprint 3 | 21 | 5 | 支付 + 订单 + 上线准备 |

## 质量原则

1. **MVP 线必须切在最基本可行的位置** — "基本"=用户能完成核心目标；"可行"=不会觉得太烂而不用
2. **一个 Story 一个 Sprint** — 一个 Story 如果需要跨 Sprint，说明拆得还不够小
3. **Story 不是任务拆分，是价值拆分** — "做登录按钮"不是 Story，"新用户能用手机号注册"才是
4. **排优先级不是排谁嗓门大** — 用价值/成本比，不是用"老板说了"
5. **假设要显式标注** — 如果一个 Story 的前提是"用户会用"，这是假设，要验证

## 使用示例

**用户输入：**
> 我们有个二手交易 App，想优化卖家发布流程。现在有很多需求：多图上传、视频、AI 自动生成描述、一键复制上次发布、定价建议、草稿保存、模板... 我不知道先做哪些。

**AI 执行过程：**
1. 画骨架：卖家发布流程 = 拍照→描述→定价→发布→管理
2. 填故事，排优先级，切 MVP 线
3. MVP = 多图上传 + 文字描述 + 基础定价 + 发布。这 5 个 Story 就够了
4. AI 自动描述、视频等到 R2/R3
5. 输出完整故事地图 + Sprint 规划
