# Performance Review Designer — 绩效体系设计

设计科学的绩效管理体系。好的绩效管理不是为了打分和排名，而是帮助每个人知道：
1) 公司需要我做什么 2) 我做得怎么样 3) 我如何做得更好。

## 何时触发

用户提及以下内容时触发：
- "帮我设计一套绩效方案"
- "OKR 怎么定？"
- "怎么做 360 度反馈？"
- "我的团队绩效面谈总是走过场，怎么办？"
- "员工表现不好，怎么做 PIP？"
- "Design a performance review process"

**排除：** 此技能 NOT for：
- 薪酬调整决策 → 独立技能（不在本技能范围）
- 裁员沟通 → 用 `employee-relations`
- 招聘评估 → 用 `structured-interview`

## 工作模式

### 模式 A：从零搭建绩效体系
为没有正式绩效管理体系的团队设计完整方案。包含：周期设计 → 目标框架 → 评估方式 → 面谈流程 → 校准机制。

### 模式 B：现有体系诊断与优化
用户描述现有体系，你诊断痛点并给出改进方案（如从 KPI 升级到 OKR+KPI 混合模式）。

### 模式 C：单项深挖
用户只关注某一环节（如只设计 PIP 模板、只设计 360 反馈问卷），输出该环节的深度方案。

## 绩效框架选择

根据团队特征推荐合适的框架：

| 团队特征 | 推荐框架 | 理由 |
|---------|---------|------|
| <20 人、变化快 | 轻量 OKR | 太重的流程会拖慢速度 |
| 20-100 人、有明确业务目标 | OKR + 价值观评估 | 兼顾结果和过程 |
| >100 人、业务稳定 | OKR + KPI + 360 | 需要更多维度保证公平 |
| 销售/客服团队 | KPI 为主 + OKR 为辅 | 量化指标天然适配 |
| 研发/创意团队 | OKR 为主 + peer review | 避免 KPI 扼杀创新 |

## 绩效管理循环

### 阶段 1：目标设定（周期初）

**OKR 设计原则：**
- O（目标）：定性、鼓舞人心、有时限（季度）
- KR（关键结果）：定量、可衡量、有挑战（信心指数 5/10）
- 每个周期 3-5 个 O，每个 O 配 2-4 个 KR

**OKR 健康检查清单：**
- [ ] 如果只看 KR，不看 O，你能知道这个人在做什么吗？
- [ ] KR 是结果（outcome）而非任务（output）？
- [ ] 80% 的 OKR 来自个人提案（而非上级下放）？

### 阶段 2：持续反馈（周期中）

**1-on-1 框架（每周/双周 30 分钟）：**
```
1. 过去两周最有成就感的事？
2. 有什么卡住了？
3. 我需要什么支持？
4. （manager）给出 1 条具体反馈
```

### 阶段 3：正式评估（周期末）

**360 度反馈问卷模板（5-8 个开放问题）：**
```
1. 此人在过去半年最大的贡献是什么？
2. 此人在哪些方面可以做得更好？
3. 如果用三个词描述此人的工作风格，你会选什么？
4. 如果此人想晋升到下一个级别，最需要提升的能力是什么？
5. （可选）还有什么想分享的？
```

### 阶段 4：绩效面谈

**面谈结构（45-60 分钟）：**
```
1. 开场（2 min）：这次面谈的目的（不是审判，是帮助成长）
2. 自我评估（10 min）："你觉得过去半年自己做得怎么样？"
3. Manager 反馈（15 min）：先说优点（具体案例），再说改进空间
4. 共识与分歧（10 min）：哪些达成一致？哪些有分歧？
5. 下周期规划（10 min）：下个周期的目标和成长方向
6. 总结与跟进（3 min）：书面记录、行动项、下次 check-in 时间
```

### 阶段 5：PIP（绩效改进计划）

**触发条件（AND 关系）：**
- ✓ 绩效持续低于预期 ≥ 1 个评估周期
- ✓ 已进行 ≥ 2 次 1-on-1 反馈
- ✓ 问题不是外部因素导致（如资源不足、角色变更）

**PIP 模板（30-60-90 天）：**
```markdown
# 绩效改进计划 — [员工姓名]

## 背景
[简述绩效差距，引用具体案例，避免主观判断]

## 改进目标（SMART）
| 目标 | 衡量标准 | 截止日期 | 权重 |
|------|---------|---------|------|
| [目标1] | [如何衡量] | [日期] | [%] |

## 支持计划
- Manager 将提供：[具体支持]
- 公司资源：[培训/mentor/工具]

## Check-in 计划
- 第 30 天：[日期] — 检查点
- 第 60 天：[日期] — 中期评估
- 第 90 天：[日期] — 最终评估

## 结果
- □ 达到预期 → 结束 PIP
- □ 部分改善 → 延长 30 天
- □ 未改善 → [后续步骤]
```

### 阶段 6：校准会议（Calibration）

**流程（适用于 >50 人团队）：**
1. 各 manager 提交评分
2. 按级别分组校准（同级对比）
3. 识别异常（同级别某人评分显著偏高/偏低 → 讨论原因）
4. 调整评分分布（非强制排名，而是确保标准一致）
5. 输出最终评估结果

## 输出模板

根据用户需求选择模式 A/B/C，输出对应方案。通用输出结构：

```markdown
# [团队名称] — 绩效管理方案

## 推荐框架
[OKR / KPI / 混合] — 理由：[基于团队特征的分析]

## 绩效周期
[季度/半年/年度]，关键节点：[日期]

## 目标设定流程
[自下而上提案 → 对齐讨论 → 最终确认]

## 评估方式
[自评 30% + Manager 评估 50% + Peer 20% / 其他]

## 面谈标准流程
[阶段 1-4 的时间分配和话术]

## 校准机制
[频率 / 参与人 / 流程]

## 实施路线图
| 时间 | 行动 | 负责人 |
|------|------|--------|
| 第 1-2 周 | [行动] | [人] |
```

## 质量原则

1. **反馈必须具体** — "你做得很好" 不是反馈，"你在 Q2 独立推进了 X 项目并在截止日前上线" 才是
2. **绩效面谈不是 surprise party** — 如果员工在正式评估时才第一次听到负面反馈，manager 失职
3. **PIP 是帮助改进的工具，不是解雇的前奏** — PIP 成功率达 50-70% 才是健康的
4. **目标自下而上** — 如果所有 OKR 都是老板定的，不是 OKR，是任务分配
5. **校准会议确保公平而非强制排名** — 如果同级别所有人都是"优秀"，可能是标准问题，不是人的问题

## 使用示例

**用户输入：**
> 我们是一个 30 人的 SaaS 创业团队，之前没做过绩效，CEO 觉得需要搞起来了。技术+产品为主。

**AI 执行过程：**
1. 追问当前痛点和绩效文化成熟度
2. 推荐：轻量 OKR + 季度评估（创业团队不适合重流程）
3. 输出模式 A：目标设定模板 → 1-on-1 框架 → 评估面谈话术 → 校准流程
4. 特别提醒：创业团队的 OKR 建议 6 周一周期（而非季度），因为变化快
