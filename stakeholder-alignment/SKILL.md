# Stakeholder Alignment — 跨部门对齐

PM 的 60% 时间不在写需求，在"对齐"。
和工程对齐可行性、和设计对齐体验、和市场对齐信息、和高管对齐优先级。
最怕的不开发做不出来——是"三个月后发现做的是 XX 部门以为的而不是 YY 部门需要的"。

## 何时触发

用户提及以下内容时触发：
- "老板又提了一个需求，怎么处理？"
- "销售和市场想要的方向完全不一样"
- "工程师说这个太难做，但我答应了客户"
- "跨部门项目怎么推？"
- "需求评审又被毙了，怎么改进？"
- "Managing stakeholders as a PM"

**排除：** 此技能 NOT for：
- 产品战略制定 → 用 `product-strategy`
- 客户 QBR → 用 `qbr-narrative`

## 开始之前

| # | 信息 | 示例 |
|---|------|------|
| 1 | **当前需要对齐的决策** | "Q3 先做 A 方案还是 B 方案" |
| 2 | **涉及的利益相关者** | "CTO（推 A）、CMO（推 B）、CEO（还没表态）" |
| 3 | **当前的僵局** | "各说各的，没人让步，项目停滞 2 周了" |
| 4 | **决策截止日期** | "下周必须定" |

## 利益相关者地图

### 四象限分析法

```
             权力大
               ↑
  "保持满意"    │    "密切管理"
  （告知、汇报） │   （对齐、协作、影响）
               │
 被动 ─────────┼──────────→ 主动
               │
  "监控"       │    "保持知情"
  （低维护）    │   （定期同步）
               │
             权力小
```

| 象限 | 策略 | 沟通频率 |
|------|------|---------|
| 权力大+主动 | **密切管理**：高频对齐，拉入核心决策圈 | 每周 1-on-1 |
| 权力大+被动 | **保持满意**：让他们知道进展，不做 surprise | 双周邮件/简报 |
| 权力小+主动 | **保持知情**：利用他们的热情，但不过度消耗你的时间 | 拉入项目群，自由参与 |
| 权力小+被动 | **监控**：低维护，有变动时通知 | 月度 newsletter |

## 常见场景与对策

### 场景 A：老板/HiPPO 提需求

> "我觉得我们应该加一个 XX 功能。"

**不要说什么：** ❌ "这不是我们这季度的优先级"（老板听到的是"你不想干"）

**应该做的：**
1. 先接住："好问题。我能先了解一下背后的动机吗？"（有时候老板只是随口提，不是真想要）
2. 如果老板是真想要 → 放进 Backlog，在优先级会中和所有想做的 Story 一起排序
3. "如果我们做 XX，就得推迟 YY 或 ZZ。你觉得呢？"（让老板参与 trade-off 决策）

### 场景 B：工程和设计的方案分歧

> 设计：要有流畅的动效。工程：动效太费时间，先做功能。

**解决框架：**
1. 各自说清楚为什么："动效不只是好看，它让用户知道系统在响应" vs "动效需要额外 3 周，延迟上线"
2. 量化 trade-off：延迟 3 周 vs 用户满意度可能提升 5 分
3. 找中间方案：能不能先没有动效上线，Sprint 3 再补？或者只做 1-2 个关键的动效？
4. PM 的角色 = 翻译 + 仲裁（不是偏袒任何一方）

### 场景 C：销售团队要求定制化

**回应对框架：**
```
"我理解这个客户对你很重要，让我们看看怎么平衡。

这个定制功能的成本是 [X 周开发 + Y 周维护]。
如果做，以下至少一个事要推迟：[列出影响]。

我们可以提供的替代方案是 [workaround / 下个版本的承诺 / 帮客户换个方式实现]。

你觉得哪种方式最好？"
```

## 对齐会议设计

### 决策前 ≠ 决策后

| 阶段 | 会议目的 | 做什么 |
|------|---------|--------|
| 决策前 | 收集输入、理解约束 | 1-on-1 和各 stakeholder 先聊 |
| 决策中 | 做决定 | 把所有人拉到一起，明确 trade-off，当场拍板 |
| 决策后 | 传达和承诺 | 书面记录决策 + 每位 stakeholder 确认理解 |

> 最大的错误：在大会上第一次暴露信息。大会是用来确认而不是讨论的。

### 决策前 1-on-1 话术

```
"[Stakeholder Name]，关于 [决策]，我想听听你的想法。

我的理解是 [你的理解]，你们团队的约束是 [约束]。
你觉得哪个方案更合理？为什么？

如果我选 A 方案，对你团队的影响是什么？"
```

收集完所有人的意见后，做一张对比表在大会上展示：

| | A 方案 | B 方案 |
|---|--------|--------|
| CTO 的意见 | 支持（可行性高） | 担心（依赖外部团队） |
| CMO 的意见 | 担心（工期长） | 支持（快） |
| 对用户的影响 | [X] | [Y] |
| **推荐** | | ✅ |

## 质量原则

1. **没有数据的分歧靠猜 → 做一个小实验** — "我们先不做完整版，花一周做个 prototype 看用户反应"
2. **大会之前先开小会** — 如果在大会上第一次听到某人的反对意见，是你准备不充分
3. **决策要有书面记录** — 因为两周后所有人都会忘记为什么选了 A 而不是 B
4. **HIppo（最高薪的人的意见）不能自动赢** — 用数据和 trade-off 讨论，而不是比谁级别高
5. **没有"所有人都同意"的方案** — 好的决策是"所有人都被听到，但不需要所有人都同意"

## 使用示例

**用户输入：**
> 我是电商 App 的 PM。CTO 想重构后端（3 个月），CMO 想做大促功能（1 个月），Q3 只能做一件。我该怎么做？

**AI 执行过程：**
1. 画利益相关者地图：CTO（权力大+主动）、CMO（权力大+主动）、CEO（权力大+还不知道）
2. 建议先和双方单独 1-on-1（而不是直接开大会）
3. 量化 trade-off：重构 → 长期降低维护成本但 Q3 没新功能上线 vs 大促 → Q3 营收增长但技术债加重
4. 输出对齐方案 + 决策框架 + CEO 汇报建议
